# Git 提交消息规范

## 语言

默认生成中文提交消息。如用户指定其他语言（如 `--lang=en`），则按指定语言生成。

## 格式

采用约定式提交（Conventional Commits）：

```
<type>[可选 scope]: <描述>

[可选正文]

[可选脚注]
```

## 核心原则：描述"为什么"而非"做了什么"

提交描述应侧重变更的动机和原因，而非仅仅罗列改了什么。

```
# 不好 — 只说了做了什么
fix: 修改登录接口返回值

# 好 — 说明了为什么改
fix: 登录接口需返回用户角色以支持权限控制

# 不好
feat: 添加缓存模块

# 好
feat: 引入缓存层以降低数据库查询延迟
```

## 提交类型

| 类型       | 用途                        |
| ---------- | --------------------------- |
| `feat`     | 新功能                      |
| `fix`      | 修复缺陷                    |
| `docs`     | 仅文档变更                  |
| `style`    | 代码格式/风格（不影响逻辑） |
| `refactor` | 重构（非功能/非修复）       |
| `perf`     | 性能优化                    |
| `test`     | 新增/更新测试               |
| `build`    | 构建系统/依赖变更           |
| `ci`       | CI/配置变更                 |
| `chore`    | 维护/杂项                   |
| `revert`   | 回退提交                    |

## 破坏性变更

使用 `!` 标记或 `BREAKING CHANGE` 脚注：

```
feat!: 移除已废弃的接口以简化 API 维护

feat: 支持配置继承以减少重复配置

BREAKING CHANGE: `extends` 键的行为已变更
```

## 正文规范

- 解释变更的动机和背景，对比前后行为
- 使用项目符号列举关键点
- 引用相关议题：`Closes #123`、`Refs #456`

## 写作要求

- 现在时态、祈使语气："添加" 而非 "添加了"
- 描述保持在 72 字符以内
- 一个提交只包含一个逻辑变更
